# Wallet API

## Описание проекта

Высокопроизводительное API для управления электронными кошельками, построенное на **FastAPI** и **SQLAlchemy** с поддержкой асинхронных операций. Система разработана для обеспечения целостности данных при высоких конкурентных нагрузках благодаря использованию блокировок на уровне базы данных (`SELECT FOR UPDATE`).

## Основные возможности

* **Управление кошельками:** Создание нового кошелька и получение его баланса.
* **Финансовые операции:** Выполнение депозитов и снятие средств.
* **Конкурентность:** Гарантия целостности данных при параллельных запросах к одному кошельку за счет пессимистичной блокировки строк в БД.
* **Асинхронность:** Полностью асинхронный код на стороне API и при работе с базой данных для максимальной производительности.
* **Миграции БД:** Управление схемой базы данных с помощью **Alembic**.
* **Контейнеризация:** Готовность к запуску в **Docker**.
* **Тестирование:** Набор интеграционных тестов для проверки корректности работы и конкурентных сценариев.

## Технологический стек

* **Язык:** Python 3.11+
* **Веб-фреймворк:** FastAPI
* **База данных:** PostgreSQL (через asyncpg)
* **ORM:** SQLAlchemy 2.0+ (asynchronous)
* **Миграции:** Alembic
* **Валидация:** Pydantic
* **Сервер:** Uvicorn
* **Тестирование:** Pytest, HTTPX, pytest-asyncio

## Быстрый старт

### 1. Запуск с помощью Docker Compose (рекомендуется)

Это самый простой способ запустить приложение со всеми зависимостями.

1. **Клонируйте репозиторий:**

    ```bash
    git clone <your-repository-url>
    cd <repository-directory>
    ```

2. **Запустите сервисы:**

    ```bash
    docker-compose up --build
    ```

Эта команда соберет образ для API, запустит контейнер с PostgreSQL, применит миграции и запустит сервер.

3.**Доступ к API:**

API доступно по адресу: [http://localhost:8000](http://localhost:8000)

Интерактивная документация Swagger: [http://localhost:8000/docs](http://localhost:8000/docs)

Альтернативная документация ReDoc: [http://localhost:8000/redoc](http://localhost:8000/redoc)

### 2. Локальный запуск (без Docker)

1. **Убедитесь, что у вас запущен PostgreSQL. Создайте базу данных (например, wallet_db).**

2. **Склонируйте репозиторий и перейдите в папку проекта.**

3. **Создайте и активируйте виртуальное окружение:**

   ```bash
    python -m venv venv
    source venv/bin/activate  # Linux/MacOS
    venv\Scripts\activate  # Windows
    ```

4. **Установите зависимости:**

    ```bash
    pip install -r requirements.txt
    ```

5. **Настройте переменные окружения:**
    Создайте файл .env в корне проекта или экспортируйте переменную DATABASE_URL.
Пример содержимого .env:

    ```env
    DATABASE_URL=postgresql+asyncpg://user:password@localhost:5432/wallet_db
    ```

    (Замените user, password, localhost и wallet_db на ваши актуальные данные)

6. **Примените миграции базы данных:**

   ```bash
   uvicorn app.main:app --reload
   ```

7. **Запустите сервер:**

     ```bash
   uvicorn app.main:app --reload
   ```

    API будет доступен по адресу: [http://localhost:8000](http://localhost:8000)

### API Endpoints

1. **Создание кошелька**

    Создает новый кошелек с нулевым балансом.
    * **URL:** /api/v1/wallets

    * **Метод:** POST

    * **Success Response:** 201 Created

        ```json
        {
        "uuid": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
        "balance": "0.00"
        }
        ```

2. **Получение баланса кошелька**

   Возвращает информацию о кошельке по его UUID.
   * **URL:** /api/v1/wallets/{wallet_uuid}

   * **Метод:** GET

   * **Success Response:** 200 OK

        ```json
        {
        "uuid": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
        "balance": "150.75"
        }
        ```

   * **Error Response:** 404 Not Found (если кошелек не существует)
  
3. **Выполнение операции (депозит/снятие)**

   Выполняет финансовую операцию над указанным кошельком.

   * **URL:** /api/v1/wallets/{wallet_uuid}/operation

   * **Метод:** POST

   * **Тело запроса (JSON):**

        ```json
        {
        "operation_type": "DEPOSIT",  // или "WITHDRAW"
        "amount": "100.50"
        }
        ```

   * **Success Response:** 200 OK

        ```json
        {
        "uuid": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
        "balance": "250.75",
        "operation_type": "DEPOSIT",
        "amount": "100.50"
        }
        ```

   * **Error Responses:**

     * *400 Bad Request:* Недостаточно средств для снятия или неверный формат суммы.

     * *404 Not Found:* Кошелек не найден.

     * *409 Conflict:* Конфликт при параллельном изменении данных (повторите запрос).

### Тестирование

Проект содержит набор интеграционных тестов, включая тесты на конкурентные операции.

## Запуск тестов в Docker Compose

Тесты используют отдельную тестовую базу данных (test_wallet_db), которая должна быть доступна

1. Убедитесь, что основной стек (docker-compose up) не запущен, или остановите его, чтобы освободить порты.
2. Запустите только базу данных для тестов:

   ```bash
   docker-compose up -d db
   ```

3. Выполните тесты: Вам нужно будет запустить отдельный контейнер или выполнить тесты локально, убедившись, что переменная TEST_DATABASE_URL в tests/conftest.py указывает на ваш тестовый БД (по умолчанию postgresql+asyncpg://user:password@db:5432/test_wallet_db). Самый простой способ - запустить тесты из вашей локальной среды, пока контейнер с БД работает:

    ```bash
   pytest tests/ -v -s
   ```

## Ручное создание тестовой БД (если необходимо)

Если вы запускаете тесты локально без Docker Compose, убедитесь, что PostgreSQL запущен и создана тестовая база данных:

```sql
CREATE DATABASE test_wallet_db;
```

Затем обновите TEST_DATABASE_URL в tests/conftest.py под ваше локальное подключение.

### Архитектура и особенности реализации

## Обработка конкурентных запросов

1. **Пессимистичная блокировка:** В методе update_balance сервиса WalletService используется конструкция .with_for_update() при выполнении запроса SELECT. Это заставляет базу данных заблокировать строку кошелька до завершения транзакции.

2. **Транзакционность:** Вся операция (чтение баланса, проверка достаточности средств, обновление) выполняется в рамках одной транзакции базы данных (async with self.db.begin():).

3. **Обработка конфликтов:** Если второй запрос попытается прочитать ту же заблокированную строку, он будет ждать снятия блокировки. Если ожидание приводит к взаимоблокировке (deadlock) или таймауту, SQLAlchemy вызовет исключение, которое перехватывается в эндпоинте и преобразуется в HTTP-ответ с кодом 409 Conflict. Клиент может повторить такой запрос.
